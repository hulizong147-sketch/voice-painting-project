# AI 语音绘图工具设计文档

## 1. 项目背景

题目要求开发一款纯语音控制的绘图工具，用户不能依赖鼠标或键盘完成绘图创作，需要通过语音指令完成绘制、修改、选择、布局、撤销、导出等操作。项目重点不只是“能画图”，还要综合考虑以下问题：

- 语音识别是否足够准确，识别错误时是否有容错能力。
- 自然语言指令能否转换成明确的绘图操作。
- 指令从语音输入到画布变化的响应延迟是否可接受。
- 复杂指令能否拆解成多个步骤执行。
- 用户能否在不使用鼠标键盘的情况下完成一段完整创作流程。

本项目最终实现为 VoiceDraw，一个基于浏览器的 AI 语音绘图工作台。

## 2. 设计目标

### 2.1 核心目标

1. 用户通过语音创建图形、文字、模板和 AI 草稿。
2. 用户通过语音选择、移动、缩放、旋转、对齐、删除和调整对象。
3. 用户通过语音设置颜色、画笔粗细、透明度、画风和画布尺寸。
4. 用户可以发出多步复杂指令，系统自动拆解并顺序执行。
5. 系统提供文字和语音反馈，让用户知道命令是否成功。
6. 系统保留历史、撤销重做和导出能力，形成完整创作闭环。

### 2.2 体验目标

- 常见命令响应尽量控制在 1 秒内。
- AI 生图类命令允许更长等待，但需要明确显示“正在生成”状态。
- 语音识别失败时能给出提示，而不是静默失败。
- 对口语化表达有一定容错，例如“画三个三角形”“来个房子”“给它戴帽子”。
- 界面虽然提供按钮用于调试和演示，但核心能力均可由语音指令触发。

## 3. 技术方案

### 3.1 总体架构

```text
用户语音
  -> 浏览器麦克风采集
  -> 百度 ASR / Web Speech API
  -> 中文命令解析器
  -> DrawingCommand 结构化指令
  -> Fabric.js 画布执行器
  -> 画布更新 + 文字反馈 + TTS 播报
```

### 3.2 前端

- React + TypeScript + Vite 构建应用。
- Fabric.js 管理画布对象、路径、图片、选择和导出。
- Zustand 保存当前颜色、描边、画风、选中数量、网格、缩放、语音状态等上下文。
- lucide-react 提供工具栏图标。

### 3.3 语音识别

项目实现了两条语音识别路径：

- 优先使用百度短语音识别：浏览器采集音频，转成 16kHz wav，经本地 Node 代理发送到百度接口。
- 浏览器 Web Speech API fallback：当百度或麦克风采集不可用时，退回浏览器内置语音识别。

语音模式支持：

- 持续监听：检测到说话和静音后自动切分语音片段。
- 按住说话：更适合录制 demo 和减少环境噪声影响。

### 3.4 指令理解

系统使用本地规则 NLU，将中文语音文本解析成统一的 `DrawingCommand` 类型。这样做的优点是延迟低、可控、便于比赛演示；缺点是泛化能力不如大模型。

解析能力包括：

- 形状识别：圆、矩形、三角形、线、星星、文字。
- 属性识别：颜色、大小、透明度、描边、画笔粗细。
- 数量识别：一个、两个、三个、5 个等。
- 位置识别：左上、右上、中间、最左边、最右边等。
- 操作识别：选择、删除、移动、缩放、旋转、翻转、对齐、分布、编组、锁定、隐藏。
- 画风识别：默认、二次元、水墨、简笔。
- AI 绘图识别：普通生成草稿、二次元人物、局部修改。
- 多命令拆解：按“然后、接着、并且、同时”等连接词切分。

### 3.5 AI 草稿生成

项目接入 Right Code Draw API 作为主要 AI 生图接口，并保留 OpenAI Images API 作为备用接口。语音命令如“画一个松鼠”“画一个二次元少女头像”会生成 AI 草稿图片并放入画布。

早期方案尝试“AI 生成草稿 + 画笔路径复刻”，即对 AI 图片进行轮廓追踪，再转成 Fabric Path。但实际效果存在断线、细节丢失和耗时问题。当前主流程改为直接把 AI 草稿放入画布，保证图像质量和响应可控；AI 描改测试面板仍保留轮廓追踪预览，便于展示技术探索过程。

### 3.6 局部编辑策略

对于“给它戴帽子”这类局部修改，如果直接调用图生图重绘整张图，AI 容易改变原图主体细节。项目最终采用分层叠加策略：

- 原图对象保持不变。
- 新增帽子、胡须、尾巴等独立图层。
- 通过对象边界和 prompt 上下文估算叠加位置。
- 新增对象可继续被选择、移动、缩放和删除。

这种策略更符合“只加一个元素，不改变原有细节”的用户预期。

## 4. 计划支持的指令能力

| 能力类别 | 计划能力 |
| --- | --- |
| 基础绘图 | 创建圆、矩形、三角形、线、星星、文字 |
| 属性控制 | 设置颜色、描边、画笔粗细、透明度、大小 |
| 多对象绘图 | 一次绘制多个对象，按行或列排列 |
| 对象选择 | 按颜色、形状、位置选择对象 |
| 对象编辑 | 移动、缩放、旋转、翻转、删除、复制、粘贴 |
| 布局操作 | 对齐、分布、图层前后顺序、编组 |
| 画布控制 | 网格、吸附、缩放、适应屏幕、画布尺寸、背景色 |
| 历史操作 | 撤销、重做、纠错 |
| AI 绘图 | 根据语音生成复杂草稿，如人物、动物、二次元头像 |
| 局部修改 | 在已有画面基础上添加帽子、尾巴、胡须等 |
| 语音反馈 | 命令执行后播报结果 |
| 导出保存 | PNG、SVG、JSON 工程 |

## 5. 最终实现情况

### 5.1 已实现

- 语音识别：百度 ASR + 浏览器 Web Speech fallback。
- 语音反馈：百度 TTS + 浏览器 SpeechSynthesis fallback。
- 基础图形：圆、矩形、三角形、直线、星星、文字。
- 模板素材：笑脸、柱状图、流程图、太阳、房子、女人头像等。
- 多对象绘制：支持“画三个三角形排成一排/一列”。
- 对象选择：支持按颜色、形状、可见性、左右上下位置选择。
- 对象编辑：移动、缩放、旋转、水平/垂直翻转、删除、复制、粘贴、重复。
- 编组和锁定：支持组合、取消组合、锁定、解锁、隐藏、显示。
- 布局能力：左/右/顶/底对齐，水平/垂直居中，横向/纵向分布。
- 样式能力：填充色、描边色、画笔粗细、透明度、文字大小、文字加粗。
- 画风能力：默认、二次元、水墨、简笔，并在状态栏显示当前画风。
- 画布能力：网格、吸附、缩放、适应屏幕、平移、画布尺寸、背景色。
- 历史能力：撤销、重做、纠正上一条命令。
- AI 草稿：根据 prompt 生成草稿图片并放到画布。
- 局部编辑：尾巴、耳朵、帽子、大眼睛、胡须、线条加粗。
- 多步指令：支持用连接词拆解并顺序执行。
- 导出保存：PNG、SVG、JSON 工程保存和恢复。
- 命令历史：展示、复制、重放、过滤、导入、导出、清空。

### 5.2 部分实现

- 纯语音控制：核心命令都可语音触发，但浏览器首次麦克风授权、文件选择、录屏操作仍受浏览器安全机制限制，需要用户手动操作。
- AI 画笔复刻：保留轮廓追踪预览，但主流程改为直接放置 AI 草稿图片，因为路径复刻在复杂二次元图上会断线、丢细节。
- 局部修改：采用叠加图层而非真正的局部 inpainting，能保证不破坏原图，但对帽子等元素的位置判断仍是启发式。
- 复杂语义理解：规则系统覆盖常见演示命令，但对非常自由的自然语言仍可能解析失败。

### 5.3 未完成及原因

| 未完成项 | 原因说明 |
| --- | --- |
| 完全零鼠标键盘交互 | 浏览器麦克风授权、文件打开下载等行为受安全策略限制，必须由用户确认。 |
| 通用大模型 NLU | 本项目优先保证响应速度和确定性，使用本地规则解析；大模型 NLU 会增加网络延迟和不可控性。 |
| 高质量局部 inpainting | 需要支持 mask 的图像编辑模型和更复杂的前景分割，目前接入的生图接口更适合整图生成。 |
| 复杂空间推理 | “把离三角形最近的圆移到矩形左上角”这类关系推理尚未完整实现。 |
| 离线语音识别 | 浏览器环境内集成离线 Whisper 成本较高，模型体积和推理性能不适合当前 demo。 |
| 系统化延迟 benchmark | 已做体验层面的响应优化，但未引入自动化性能测试脚本。 |

## 7. 延迟与容错设计

### 7.1 延迟控制

- 普通图形命令：本地解析和本地画布执行，通常接近即时。
- 多步命令：拆分后顺序执行，每一步仍是本地操作。
- 语音识别：百度 ASR 需要上传音频，延迟主要取决于录音长度和网络。
- AI 草稿：外部生图接口耗时不可完全控制，界面会显示生成状态。

### 7.2 容错设计

- 支持常见同义表达，如“画、绘制、生成、来个”。
- 支持数量、颜色、形状和位置词的组合解析。
- 支持命令纠正，如“不对，改成蓝色”“等一下，放大一点”。
- 百度不可用时退回浏览器语音识别。
- TTS 不可用时退回浏览器语音合成。
- AI 配置缺失时显示明确错误，避免静默失败。

## 8. Demo 建议

推荐演示路线：

1. 展示语音监听状态和当前画风状态。
2. 用语音画基础图形和多对象排列。
3. 用语音选择对象并移动、缩放、改色。
4. 用语音切换画风和画布设置。
5. 用语音生成 AI 草稿。
6. 在 AI 草稿基础上说“给它戴帽子”，展示原图不变，只新增帽子图层。
7. 用撤销、重做和导出收尾。
